# LLM-MBTI Recommender

## Problem Statement
HMW — MBTI라는 익숙한 어휘를 입구로 삼아, 사람들이 "내 직무와 작업에 맞는 LLM"을
자기 언어로 이해하고 한 모델을 골라 실제로 써보게 만들 수 있을까?

## Recommended Direction
**3단계 진단형 추천 도구**. 사용자는 (1) MBTI 16유형 중 본인을 선택하고,
(2) 직무·소속(개발자/기획자/마케터/연구자/학생/콘텐츠 크리에이터 등)을 고른 뒤,
(3) 주된 작업 유형(코드 작성·글쓰기·리서치·요약·번역·이미지 분석 등 다중 선택)과
우선순위(정확성/속도/비용/창의성)를 입력한다. 결과 화면에는 추천 LLM 1-3개와
"왜 이 조합에 이 모델인지" 설명, 바로 복사 가능한 시스템 프롬프트 샘플, 공유용
결과 카드가 나온다.

MBTI는 **친숙한 입구이자 결과 카드의 톤**으로 쓰고, 실제 추천 로직의 무게는
직무·작업·우선순위에 둔다. 이렇게 하면 MBTI의 심리측정학적 약점을 우회하면서도
"INTJ는 Claude, ENFP는 Gemini" 같은 공유 가능한 후크를 살릴 수 있다. 백엔드
추천 근거는 공개 벤치마크(MMLU·HumanEval·MT-Bench) + 각 모델 강점 매트릭스를
직무·작업 축으로 매핑한 정적 테이블로 시작한다.

포트폴리오 목적과 정합 — 3분 안에 결과를 보여줄 수 있고, OG 이미지가 있는
결과 카드는 SNS 공유에 적합해서 데모 임팩트가 크다.

## Key Assumptions to Validate
- [ ] **Must Be True** — MBTI×직무×작업의 조합이 의미 있는 모델 추천으로 수렴한다
  (n=10 직접 진단 시 추천 모델이 2-3개 안으로 좁혀지고, 사용자가 "납득된다"고 답한다).
  검증: 종이 프로토타입으로 지인 10명에게 진단, 추천이 분산되는지 / 공감되는지 인터뷰.
- [ ] **Should Be True** — 결과 카드가 공유하고 싶을 만큼 매력적이다
  (베타 사용자 중 20%+ 가 결과 카드를 공유하거나 스크린샷을 찍는다).
  검증: 결과 페이지에 명시적 "공유" CTA + OG 이미지를 붙이고 행동을 관찰.
- [ ] **Should Be True** — "추천된 시스템 프롬프트를 복사해 가는" 액션이 발생한다
  (방문자의 10%+ 가 프롬프트 복사 버튼을 누른다).
  검증: 복사 버튼 클릭 이벤트 로깅. 추천 이상의 실용 가치 신호.
- [ ] **Might Be True** — 사용자가 MBTI를 모르거나 확신이 없는 경우 이탈한다
  (간이 진단 quiz 옵션이 필요하다). MVP 후 행동 데이터로 결정.

## MVP Scope

**포함**:
- 3단계 입력 흐름 (MBTI 그리드 → 직무 단일 선택 → 작업 다중 선택 + 우선순위 슬라이더)
- 정적 추천 테이블 (MBTI×직무×작업 → 모델 1-3개) — JSON으로 관리, 12개 정도의 LLM 후보
- 결과 페이지: 추천 모델 카드 + 근거 텍스트 + 시스템 프롬프트 샘플(복사 버튼) + OG 이미지
- 결과 URL은 입력값을 쿼리스트링으로 보존 → 공유 가능
- 익명 이벤트 로깅 (단계별 이탈, 복사 클릭, 공유 클릭)

**빠짐**:
- 로그인·계정·DB
- 실제 LLM API 호출이나 라우팅 (추천만, 실행은 사용자가)
- MBTI를 모르는 사용자용 간이 진단 quiz (검증 후 추가)
- 팀·조직 단위 진단·보고서
- 모델 가격·SLA·보안 비교 (B2B 도입 담당자용 기능)

## Not Doing (and Why)
- **GitHub/로그 기반 역방향 진단** — 차별성은 높지만 OAuth + 분석 비용 +
  검증 사이클 길어짐. 포트폴리오 데모 임팩트(3분 진단) 손해.
- **16유형 전부에 대해 풍부한 카피 작성** — 처음엔 직무·작업의 조합으로 추천이
  결정되도록 하고, MBTI는 결과 카드 톤에만 영향. 16유형 카피는 사용 신호 본 뒤 확장.
- **실제 모델 라우팅·실행** — 추천 도구의 본질은 "고르기 전의 의사결정 지원".
  실행까지 가면 범위가 폭발하고 사용자가 본인 환경에서 쓸 이유가 약해짐.
- **팀 보고서·관리자 대시보드** — 타겟이 B2B로 흘러가면 검증 사이클이 길어져
  본인 학습·포트폴리오 목적과 충돌.
- **유료화·결제** — 학습 신호와 무관.

## Open Questions

> 보류 사유와 답할 시점을 함께 표기. 모든 질문을 ideate 단계에서 답할 필요는 없다.

**spec 시작 전에 답해야 하는 것 (`/write-spec` 직전):**
- 추천 매핑을 정적 테이블로 둘지, LLM 호출로 동적 생성할지 — 화면의 수용 기준을 가른다.
- 추천 후보 LLM 풀에 어떤 모델을 몇 개 넣을지 — 결과 화면의 variation 수를 결정한다.

**spec 단계(와이어프레임)에서 자연스럽게 결정되는 것:**
- 한국어 우선인지, 한·영 동시 지원인지 — 직무 카피·결과 톤이 갈린다.
- "왜 이 모델인지" 설명의 분량·구조 — 정적 템플릿이면 길이 한계, 동적이면 형식 한계.

**build/legal 단계에서 답할 것 (지금 보류):**
- 결과 카드에 각 모델 로고·브랜드 컬러를 쓸 수 있는지 (벤더 브랜드 가이드 확인 필요).
- 추천 매핑의 신선도 갱신 주기 — 모델이 자주 바뀌므로 운영 방식이 필요.
